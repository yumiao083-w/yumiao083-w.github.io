# 情感记忆检索 & yuan 项目改造方案

> 2026-03-11 与用户讨论产生，持续更新

## 核心洞察

**传统 RAG 的局限**：embedding 只做数学上的"语义相似度"匹配，不理解对话语境、情感关联、个人习惯。

**用户的独立推导**：从 embedding 概念出发，自主推导出多智能体分层记忆架构 + 情感化检索方向。

## 架构概览

### 五板块注入系统

```
┌──────────────────────────────────────────────────────┐
│                    每次必注入                          │
│  ┌─────────────┐  ┌──────────┐  ┌─────────────────┐  │
│  │ 1.人设卡     │  │ 2.预设    │  │ 3.核心记忆       │  │
│  │ character.md │  │preset.md │  │ core_memory     │  │
│  │ ~5k token   │  │ ~3k token│  │ ~5k→1-2k token  │  │
│  └─────────────┘  └──────────┘  └─────────────────┘  │
├──────────────────────────────────────────────────────┤
│                  智能注入                              │
│  ┌──────────────────┐  ┌───────────────────────────┐ │
│  │ 4.记忆索引（常驻） │  │ 5.检索命中详情（按需）     │ │
│  │ 181条 summary    │  │ 关键词匹配 top 5 展开     │ │
│  │ ~13k token       │  │ ~1k token               │ │
│  └──────────────────┘  └───────────────────────────┘ │
└──────────────────────────────────────────────────────┘

新版总计: ~27k token（旧版: ~200k+ token，节省 85%+）
实测: Claude 4.6 Opus 125k→59k / Gemini 3 87k→41k
```

### 数据格式

```json
{
  "id": 1,
  "date": "2025-09-28",
  "event": "户外安全, 徒步, 玉珠峰, 山难",
  "summary": "她突然对山难案例感兴趣...(30-60字情感概要)",
  "content": "...(完整详细记忆，检索命中后才注入)",
  "memo": "...(碎碎念备忘录)"
}
```

### 检索管道

```
用户消息 + 最近 3 轮对话
    ↓
路径选择（config: MEMORY_RETRIEVAL_MODE）
  ├─ 'llm'  → 全量 summary 丢给便宜模型（Flash/Pro）
  │           system 角色锁定为"检索系统"，不回复用户
  │           发散联想策略：话题→关联话题→情感→因果→习惯
  │           不限返回条数，AI 自己判断多少条相关
  │           失败时自动降级 keyword
  ├─ 'keyword' → event 关键词匹配（降级兜底）
  └─ 'off'  → 不检索

LLM 精筛故障转移：
  中转站 A → 超时/报错 → 中转站 B → ... → 全挂 → 降级 keyword
  连续失败 3 次的站自动冷却 120s

实测效果：
  - 徒步场景：返回 14 条相关记忆（含直接相关 + 联想关联）
  - token 增量 ~5k（66.8k 总量），可接受
```

### 未来演进方向

**记忆 >500 条时：**
```
embedding 粗筛 top 50 → LLM 精筛（不限条数）
（embedding 做缩小候选池，不做最终决策）
```

**记忆标签/分类系统（讨论中）：**
见下方"记忆标签系统设计思路"章节

## 设计原则

- **JSON 存储，自然语言注入**：代码读 JSON，AI 读 Markdown
- **event 触发，summary 精筛**：客观关键词做冷检索，主观概要做热检索
- **索引常驻，详情按需**：AI 始终看到全局概要，需要时才展开详情
- **向后兼容**：character.md 不存在时回退到 袁朗.md

## 记忆标签系统设计思路

> 2026-03-15 讨论产生

### 问题

记忆越来越多后，全量丢给 LLM 精筛的成本会上升。需要一种方式对碎片记忆做细分，方便检索时缩小范围。

但传统的**分类**（每条记忆只属于一个类别）太粗暴——"西藏徒步时吵架"同时涉及"徒步"、"旅行"、"争吵"、"情感"。

### 方案：多标签（Tags）而非分类（Categories）

每条记忆可以有 **多个标签**，标签不是硬分类而是维度标记：

```json
{
  "id": 15,
  "date": "2025-10-05",
  "tags": ["徒步", "西藏", "身体状况", "冒险", "坚韧"],
  "event": "西藏徒步, 高反, 发烧, 翻垭口",
  "summary": "...",
  "content": "..."
}
```

### 标签维度

不是按"话题"一个维度打标签，而是多个维度交叉：

| 维度 | 示例标签 |
|------|---------|
| **活动类型** | 徒步、做饭、看电影、工作、学习 |
| **地点** | 西藏、云南、昆明、家里 |
| **情感状态** | 开心、低落、争吵、感动、焦虑 |
| **关系动态** | 亲密、冷战、磨合、突破、日常 |
| **主题** | 健康、安全、食物、装备、计划 |

### 标签怎么生成

**方案 A：AI 自动打标签**
- 新增记忆时（`[SAVE_MEMORY]` 触发），让生成 entry 的那个 prompt 同时输出 tags 字段
- 已有 181 条 → 批量跑一次，让 LLM 给每条打标签

**方案 B：从 event 字段衍生**
- event 本身已经是关键词列表（`"户外安全, 徒步, 玉珠峰, 山难"`）
- 可以把 event 作为标签的基础，再补充情感/关系维度的标签

### 标签怎么用

**近期（<500 条）**：标签暂时只是元数据，全量 LLM 精筛时可以作为额外信息帮助 AI 判断相关性。

**远期（>500 条）**：
```
用户消息 → LLM 判断相关标签 → 按标签过滤候选池 → LLM 精筛
```
比如用户说"上次吵架"→ LLM 判断相关标签为 ["争吵", "冷战", "情感"] → 只把带这些标签的记忆（比如 30 条）丢给精筛，而不是全部 500+ 条。

### 要不要现在做？

**建议：现在只做标签生成（给 181 条打标签），暂不改检索管道。**
- 打标签是低成本操作（批量调一次 LLM）
- 标签存进 memory_entries.json 后不影响现有检索
- 等记忆多了再在检索管道里加标签过滤层
