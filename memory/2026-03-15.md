# 2026-03-15 工作日志

## 今日完成

### 记忆检索管道 v2（核心工作）

**背景讨论**：
- 用户分享了多个记忆系统参考方案（混合搜索+reranker、AI工具自主搜索、Markdown文件流 vs RAG、MCP记忆库等）
- 讨论了 embedding 粗筛可能丢失情感相关但语义不近的记忆的问题
- 确定方案：181条规模下，全量 summary 直接丢给便宜 LLM 精筛，不做 embedding 预筛（等 >500 条时再加）
- 用户要求中转站配置全部前端可修改，支持获取模型列表

**新建 `memory_llm_ranker.py`**：
- `_call_provider()` — 调单个中转站 Chat Completion API，带超时
- `_failover_call()` — 按顺序故障转移，跳过冷却中的站点
- `rank_memories()` — 核心精筛：全量 summary + 对话上下文 → 便宜模型选 top N
- `_parse_ranked_ids()` — 兼容多种 LLM 输出格式（纯JSON/代码块/夹杂文字）
- `fetch_provider_models()` — 获取中转站可用模型列表
- 冷却机制：连续失败 3 次自动冷却 120s

**改造 `memory_retrieval.py`**：
- `retrieve_memories()` → 路由入口（llm/keyword/off 三种模式）
- `retrieve_by_keyword()` — 原版关键词匹配独立为函数，作为 fallback
- `_get_entries_by_ids()` — 根据 id 列表获取完整条目
- LLM 失败自动降级到关键词，无感切换

**改造 `bot.py`**：
- 检索调用点传入 `recent_context`（最近 6 条消息 ≈ 3 轮对话）

**`config.py` 新增配置**：
- `MEMORY_RETRIEVAL_MODE` = 'llm'
- `MEMORY_LLM_PROVIDERS` — 多中转站列表（name/url/key/model/timeout）
- `MEMORY_RETRIEVAL_TOP_K` = 5
- `MEMORY_FALLBACK_TO_KEYWORD` = True

**`config_editor.py` 新增 API**：
- `GET/POST /api/memory_retrieval_config` — 读写检索配置
- `POST /api/memory_retrieval_fetch_models` — 获取中转站模型列表

**前端配置界面**（`config_editor.html`）：
- 记忆检索管道配置区块（在记忆功能配置页内）
- 检索模式下拉选择
- 中转站卡片列表：添加/编辑/删除/拖拽排序
- 每个站：名称、URL、Key、模型（下拉获取+手动输入）、超时
- 降级开关、检索条数
- 保存按钮 + 状态提示
